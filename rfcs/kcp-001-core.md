         Knowledge Context Protocol (KCP) — Core Specification

Status of This Memo

   This document specifies an experimental protocol for the Internet
   community, and requests discussion and suggestions for improvements.
   Distribution of this memo is unlimited.

Abstract

   This document defines the Knowledge Context Protocol (KCP), an
   application-layer protocol for persistent governance, discovery, and
   lineage tracking of knowledge outputs generated by artificial
   intelligence agents. KCP introduces a new abstraction layer (Layer 8:
   Context & Knowledge) above the OSI Application Layer (Layer 7),
   addressing the growing need for standardized management of AI-
   generated knowledge across organizational boundaries.

1.  Introduction

   The proliferation of artificial intelligence agents across enterprise
   and individual workflows has created an unprecedented volume of
   generated knowledge — analyses, reports, insights, visualizations,
   and recommendations. As of early 2026, knowledge workers generate
   40-60 analytical outputs per week using AI assistants such as large
   language models (LLMs), code generation tools, and custom agents.

   However, this knowledge is fundamentally ephemeral. Current
   infrastructure treats AI-generated outputs as disposable session
   artifacts:

   (a)  Knowledge disappears when AI sessions terminate.

   (b)  No standard mechanism exists for discovering whether similar
        analysis has been performed previously.

   (c)  No protocol tracks the provenance chain from data sources
        through queries and agents to final insights and decisions.

   (d)  No multi-tenant governance framework controls access based on
        organizational context rather than purely technical RBAC.

   The OSI Reference Model [RFC1122] defines seven layers of network
   communication, from Physical (Layer 1) to Application (Layer 7).
   The term "Layer 8" has been used informally since the 1980s to
   refer to user or political factors [SCHNEIER2010]. This document
   redefines Layer 8 with a formal technical specification: a
   protocol layer for Context and Knowledge governance.

   KCP addresses the gap between application-layer data exchange
   (Layer 7) and the emerging need for persistent knowledge governance
   by defining:

   -  A standard payload format for knowledge artifacts

   -  A protocol for publishing, discovering, and retrieving artifacts

   -  A multi-tenant governance model with tiered visibility

   -  A lineage tracking mechanism for full provenance

   -  A federated peer-to-peer architecture for cross-organizational
      knowledge sharing


2.  Terminology

   The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT",
   "SHOULD", "SHOULD NOT", "RECOMMENDED", "NOT RECOMMENDED", "MAY",
   and "OPTIONAL" in this document are to be interpreted as described
   in BCP 14 [RFC2119] [RFC8174].

   Knowledge Artifact:
      Any output generated by an AI agent or human-AI collaboration
      that represents analytical insight, including but not limited to:
      reports, analyses, visualizations, recommendations, summaries,
      and structured data extracts.

   Tenant:
      An isolated organizational context within which knowledge
      artifacts are governed. A tenant MAY represent a company,
      division, project, or any logical grouping.

   Team:
      A subgroup within a tenant. Teams provide secondary access
      control granularity within organizational hierarchies.

   Lineage:
      The directed acyclic graph (DAG) representing the provenance
      chain from data sources through processing agents to final
      knowledge artifacts and subsequent decisions.

   Visibility Tier:
      One of four access control levels: public, org, team, or private.
      See Section 6 for full definitions.

   Node:
      A KCP-compliant server or peer that stores, indexes, and serves
      knowledge artifacts.

   Federation:
      The network of interconnected KCP nodes that enables cross-
      organizational knowledge discovery.


3.  Protocol Overview

3.1.  Layer Model

   KCP operates as a Layer 8 protocol above the OSI Application Layer:

      +-------------------------------------------+
      |  Layer 8: Context & Knowledge (KCP)       |
      +-------------------------------------------+
      |  Layer 7: Application (HTTP, gRPC, etc.)  |
      +-------------------------------------------+
      |  Layers 1-6: Traditional Network Stack    |
      +-------------------------------------------+

   KCP is transport-agnostic. Implementations MAY use HTTP/2, HTTP/3,
   gRPC, or any reliable transport protocol at Layer 7.

3.2.  Core Operations

   KCP defines five core operations:

      PUBLISH   - Submit a new knowledge artifact to a node
      DISCOVER  - Search for artifacts by keywords, tags, or semantics
      RETRIEVE  - Fetch a specific artifact by identifier or content hash
      VERIFY    - Validate the integrity and authenticity of an artifact
      REVOKE    - Soft-delete an artifact (preserving lineage references)


4.  Knowledge Artifact Format

4.1.  Schema Definition

   A KCP knowledge artifact MUST contain the following fields:

   +--------------------+----------+----------------------------------+
   | Field              | Type     | Description                      |
   +--------------------+----------+----------------------------------+
   | id                 | string   | UUID v4 unique identifier        |
   | version            | string   | Protocol version ("1")           |
   | user_id            | string   | Creator's identifier             |
   | tenant_id          | string   | Organization identifier          |
   | timestamp          | string   | ISO 8601 UTC creation time       |
   | format             | string   | Content type (see 4.3)           |
   | visibility         | string   | Access tier (see Section 6)      |
   | title              | string   | Human-readable title             |
   | content_hash       | string   | SHA-256 hex of content           |
   | signature          | string   | Ed25519 signature                |
   +--------------------+----------+----------------------------------+

   A KCP knowledge artifact MAY contain the following optional fields:

   +--------------------+----------+----------------------------------+
   | Field              | Type     | Description                      |
   +--------------------+----------+----------------------------------+
   | team               | string   | Subgroup within tenant           |
   | tags               | [string] | Keywords for discovery           |
   | source             | string   | Agent identifier + version       |
   | summary            | string   | Brief description (max 500 ch.)  |
   | lineage            | object   | Provenance information (Sec. 7)  |
   | content_url        | string   | URI where content is stored      |
   | embeddings         | [float]  | Vector for semantic search       |
   | acl                | object   | Fine-grained access control      |
   +--------------------+----------+----------------------------------+

4.2.  Identifier Requirements

   The "id" field MUST be a UUID v4 as defined in [RFC4122]. Nodes
   MUST reject artifacts with duplicate identifiers with HTTP 409
   Conflict.

4.3.  Supported Formats

   The "format" field MUST be one of:

      html       - HTML document
      json       - JSON structured data
      markdown   - Markdown text
      pdf        - PDF document
      png        - PNG image
      csv        - Comma-separated values
      text       - Plain text

   Implementations MAY extend this list. Unknown formats SHOULD be
   accepted but MAY receive reduced indexing support.

4.4.  Content Integrity

   The "content_hash" field MUST contain the lowercase hexadecimal
   encoding of the SHA-256 hash of the artifact's content bytes.

   Upon retrieval, clients SHOULD verify that the SHA-256 hash of
   received content matches the "content_hash" field.

4.5.  Digital Signature

   The "signature" field MUST contain an Ed25519 [RFC8032] signature
   computed as follows:

   1. Construct a canonical JSON representation of the artifact,
      excluding the "signature" field itself.

   2. Sort all keys alphabetically at every nesting level.

   3. Use compact serialization (no whitespace).

   4. Compute Ed25519 signature over the UTF-8 bytes of this
      canonical JSON using the user's private key.

   5. Encode the signature as lowercase hexadecimal.

   Nodes MUST verify signatures upon PUBLISH and MUST reject
   artifacts with invalid signatures with HTTP 401 Unauthorized.


5.  Operations

5.1.  PUBLISH

   Request:
      POST /kcp/v1/reports
      Content-Type: application/json

      {knowledge artifact payload as defined in Section 4}

   Processing:
      1. Validate schema conformance
      2. Verify Ed25519 signature
      3. Verify tenant_id authorization
      4. Compute content_hash if not provided; verify if provided
      5. Generate embeddings (if implementation supports)
      6. Store artifact in backend
      7. Index for full-text and vector search
      8. Announce to federation (if configured)

   Response:
      201 Created
      Location: /kcp/v1/reports/{id}

      {"id": "...", "content_hash": "...", "indexed_at": "..."}

5.2.  DISCOVER

   Request:
      GET /kcp/v1/reports?q={query}&tenant_id={tenant}&team={team}
          &tags={tag1,tag2}&from={date}&to={date}&limit={n}&offset={n}

   All parameters are OPTIONAL. If no parameters are provided, the
   node SHOULD return the most recent artifacts accessible to the
   requesting user.

   Response:
      200 OK
      Content-Type: application/json

      {
        "results": [{artifact_summary}, ...],
        "total": 42,
        "query_time_ms": 18
      }

   Each artifact_summary MUST contain: id, title, summary, created_at,
   relevance (float 0-1), and format.

5.3.  RETRIEVE

   Request:
      GET /kcp/v1/reports/{id}

   Response:
      200 OK - Artifact metadata (JSON)
      404 Not Found - Artifact does not exist or is not accessible

   Content retrieval:
      GET /kcp/v1/reports/{id}/content

   Response:
      200 OK - Raw content with appropriate Content-Type header
      403 Forbidden - ACL violation

5.4.  VERIFY

   Request:
      POST /kcp/v1/verify

      {"id": "...", "content_hash": "...", "signature": "..."}

   Response:
      200 OK - {"valid": true, "verified_at": "..."}
      400 Bad Request - {"valid": false, "reason": "..."}

5.5.  REVOKE

   Request:
      DELETE /kcp/v1/reports/{id}

   Nodes MUST implement soft deletion. The artifact metadata MUST be
   retained with a "deleted_at" timestamp. Content MAY be purged after
   a configurable retention period.

   Lineage references to revoked artifacts MUST remain valid.

   Response:
      200 OK - {"deleted_at": "..."}
      404 Not Found


6.  Multi-Tenant Governance

6.1.  Visibility Tiers

   +----------+--------------------------------------------------+
   | Tier     | Access Rule                                      |
   +----------+--------------------------------------------------+
   | public   | Any authenticated or anonymous user MAY read     |
   | org      | Any user with matching tenant_id MAY read        |
   | team     | Any user with matching tenant_id AND team MAY    |
   |          | read                                             |
   | private  | Only the creator (user_id) and users explicitly  |
   |          | listed in the ACL MAY read                       |
   +----------+--------------------------------------------------+

   The "visibility" field is REQUIRED on all artifacts. Nodes MUST
   enforce visibility rules on all DISCOVER and RETRIEVE operations.

6.2.  Access Control List (ACL)

   The optional "acl" field provides fine-grained access control:

      {
        "allowed_tenants": ["string"],
        "allowed_users": ["string"],
        "allowed_teams": ["string"]
      }

   When present, the ACL OVERRIDES the visibility tier. Access is
   granted if the requesting user matches ANY of: allowed_users,
   allowed_teams, or allowed_tenants.

6.3.  Tenant Isolation

   Nodes MUST ensure cryptographic isolation between tenants:

   -  Each tenant MUST have a unique encryption key
   -  Tenant A's artifacts MUST NOT be decryptable by Tenant B
   -  Cross-tenant search MUST only return public-visibility artifacts


7.  Lineage Tracking

7.1.  Lineage Object

   The optional "lineage" field records provenance:

      {
        "query": "string",
        "data_sources": ["uri"],
        "agent": "string",
        "parent_reports": ["uuid"]
      }

   -  "query": Human-readable description of what question was answered
   -  "data_sources": URIs identifying input data (databases, APIs, etc.)
   -  "agent": Identifier of the AI agent or tool that generated the
      artifact
   -  "parent_reports": UUIDs of artifacts this one builds upon

7.2.  Lineage Traversal

   By following "parent_reports" chains, clients can reconstruct full
   provenance graphs. This graph MUST be a Directed Acyclic Graph (DAG).

   Nodes SHOULD provide a convenience endpoint for lineage traversal:

      GET /kcp/v1/reports/{id}/lineage?depth={n}

   Response: DAG of artifacts up to {n} levels deep.


8.  Discovery & Search

8.1.  Full-Text Search

   Nodes MUST implement full-text search across the "title", "summary",
   and "tags" fields. Implementations SHOULD use inverted indexes
   (e.g., FTS5 for SQLite-based backends).

8.2.  Semantic Search (OPTIONAL)

   Nodes MAY implement vector-based semantic search using the
   "embeddings" field. When supported:

   1. Client generates embedding for search query
   2. Node performs approximate nearest neighbor (ANN) search
   3. Results ranked by cosine similarity

8.3.  Result Ranking

   When both full-text and semantic search are available, nodes
   SHOULD merge results using Reciprocal Rank Fusion (RRF):

      RRF_score(d) = Σ 1 / (k + rank_i(d))

   where k is a constant (RECOMMENDED: k=60) and rank_i(d) is the
   rank of document d in result set i.


9.  Security Considerations

9.1.  Transport Security

   All KCP communication MUST use TLS 1.3 [RFC8446] or later.
   Implementations MUST NOT fall back to unencrypted transport.

9.2.  At-Rest Encryption

   Nodes MUST encrypt stored artifacts using AES-256-GCM [RFC5116].
   Encryption keys MUST be unique per tenant (see Section 6.3).

9.3.  Digital Signatures

   All artifacts MUST be signed using Ed25519 [RFC8032]. Nodes MUST
   verify signatures on PUBLISH and SHOULD verify on RETRIEVE.

   Key management:
   -  Each user MUST have a unique Ed25519 keypair
   -  Private keys MUST NOT be transmitted or stored on nodes
   -  Public keys MUST be discoverable (via user profile endpoint
      or out-of-band exchange)

9.4.  Replay Protection

   Nodes MUST reject artifacts with timestamps more than 5 minutes
   in the past or future (accounting for clock skew). Implementations
   MAY use a configurable tolerance window.

9.5.  Rate Limiting

   Nodes MUST implement rate limiting per user_id and per tenant_id.
   RECOMMENDED limits:
   -  PUBLISH: 100 requests/minute per user
   -  DISCOVER: 1000 requests/minute per user
   -  RETRIEVE: 500 requests/minute per user

9.6.  Key Hierarchy

   RECOMMENDED key hierarchy:

      Tenant Root Key (AES-256)
      ├── Storage Encryption Key (derived via HKDF)
      ├── Team Keys (derived via HKDF + team identifier)
      └── User Signing Keys (Ed25519, independent)


10.  Storage Layer

10.1.  Storage Abstraction

   KCP defines a storage interface, not a specific backend.
   Implementations MUST support:

   -  store(artifact) → content_hash
   -  retrieve(content_hash) → artifact_content
   -  delete(id) → boolean
   -  list(filters) → [artifact_metadata]
   -  sync(peer) → sync_result

10.2.  Reference Backends

   This specification describes three reference backends:

   (a)  libsql + SQLCipher: Embedded SQL database with encryption.
        Suitable for single-node deployments and MVPs.

   (b)  IPFS + libp2p: Content-addressed P2P storage. Suitable for
        federated deployments.

   (c)  KCP Native Format: Append-only log file (.kcp extension).
        Self-contained, portable, verifiable. Suitable for offline
        sharing and archival.


11.  Federation

11.1.  Node Types

   +------------+----------------------------------------------------+
   | Type       | Description                                        |
   +------------+----------------------------------------------------+
   | Full Node  | Stores artifacts, participates in DHT, serves API  |
   | Light Node | Queries DHT, stores locally only                   |
   | Relay Node | Assists NAT traversal, does not store artifacts     |
   +------------+----------------------------------------------------+

11.2.  Peer Discovery

   Federated nodes MUST use Kademlia DHT [MAYMOUNKOV2002] for peer
   discovery. Each node announces:

   -  Its tenant_id
   -  Available tag categories
   -  Node type and capabilities

11.3.  Sync Protocol

   Nodes SHOULD sync using Merkle DAG diff:

   1. Exchange Merkle tree roots
   2. Identify differing subtrees
   3. Transfer only changed artifacts
   4. Verify signatures and hashes post-transfer

11.4.  Conflict Resolution

   KCP uses append-only semantics. Conflicts are impossible by design:

   -  Artifacts are immutable once published
   -  Updates create new artifacts with parent_reports references
   -  Deletes are soft (tombstone markers)
   -  No merge operations exist
